3 - Transporte

Protocolos de transporte

El nivel de transporte o capa de transporte es el cuarto nivel del modelo OSI,1 y está encargado de la transferencia libre de errores de los datos entre el emisor y el receptor, aunque no estén directamente conectados, así como de mantener el flujo de la red. Es la base de toda la jerarquía de protocolo.

La tarea de esta capa es proporcionar un transporte de datos eficiente y económico de la máquina de origen a la máquina destino (Host a Host) independientemente de las de redes físicas en uno.

Principales protocolos:

  • TCP
  • UDP

Host a Host

Se caracteriza por realizar el transporte de contenido (data) de un host a otro. El contenido es información provista por la capa superior, que es la capa de aplicación.

Mas precisamente, un mensaje HTTP es transportado por un mensaje de (en el caso de la image) un mensaje TCP.

Es decir, existe un encapsulamiento de la información.

Puertos

En un host habrán multiples aplicaciones que deseen enviar información a otros hosts. Cada uno enviara mensajes a hosts diferentes, los cuales serán identificados por una dirección IP que se encargara la capa de red (inferior a transporte). Sin embargo, ante multiples paquetes entrantes, cada uno sera para una aplicación. El problema esta en distinguir, dentro de un hosts, quien debe recibir cada paquete.

El protocolo de transporte sera el encargado de manejar esto, y lo hara mediante el uso de puertos.

Cuando una aplicación desea enviar un mensaje, lo debe hacer mediante un puerto y este debe ser único para cada aplicación.

Por ejemplo, un programa X decide abrir el puerto 3200. Se lo solicita al Sistema Operativo, y este puede decidir otorgarle o no dicho puerto (verificando permisos y que dicho puerto este libre). Luego puede utilizar este puerto para enviar y/o recibir información utilizando dicho puerto. Cuando llegue información al puerto 3200, la capa de transporte derivara a la aplicación que utilice el puerto 3200, el mensaje contenido en la sección Data.

UDP

Es un protocolo:

  • No confiable
    • No utiliza ACK - Acknowledge (Confirmación). Por lo que no sabe si se ha recibido un mensajes
  • No orientado a conexión
    • Sin control de flujo (TCP lo implementa)

El mensaje que se la capa de aplicación decidió enviar, se encapsula en la sección de data dentro de un mensaje UDP.

  • Puerto destino
    • Determina que puerto debera recibir la informacion en el otro host, es decir, que apliacion debe leer la informacion
  • Puerto origen
    • Es opcional, pero es util para que el host destino pueda determinar a que puerto responder del host en caso de que lo desee.
  • Longitud del mensaje
    • Simplemente el tamanio total del mensaje
  • Checksum (Suma de verificación)
    • Tambien es opcional, y sirve para validar la integridad del contenido.

Checksum

Es una función de redundancia que tiene como propósito principal detectar cambios accidentales en una secuencia de datos para proteger la integridad de estos, verificando que no haya discrepancias entre los valores obtenidos al hacer una comprobación inicial y otra final tras la transmisión. La idea es que se transmita el dato junto con su valor suma, de esta forma el receptor puede calcular dicho valor y compararlo así con el valor suma recibido. Si hay una discrepancia se pueden rechazar los datos o pedir una retransmisión.

Objetivo → Determinar si la información recibida sufrió alteraciones para tomarla como valida o no.

TCP

El objetivo de TCP es crear conexiones dentro de una red de datos compuesta por redes de computadoras para intercambiar datos. Además, en cuanto a su funcionamiento, garantiza que los datos llegarán a destino sin errores y en el mismo orden en el que fueron transmitidos. Como también lo hace UDP, proporciona un mecanismo para distinguir distintas aplicaciones dentro de una misma máquina, a través del concepto de puerto.

TCP clasificara cada mensaje por tipos, los cuales se especificaran en un flag dentro del mensaje. Existen los siguientes tipos de mensajes:

  • SYN - Inicializar conexión
  • ACK - Confirma recepción de información
  • FIN - Cerrar conexión
  • RST- Aborta una conexión en respuesta a un error.

Conexión - Negociación en tres pasos

Dos hosts, en este caso cliente y servidor, establecerán una conexión inicializada por el cliente.

El cliente enviara un mensaje TCP con el flag SYN indicando que se desea iniciar la conexión.

Aquí el servidor puede determinar si acepta o no la conversación. En caso de aceptar, responde con ambos flags SYN-ACK indicando que acepta establecer la conexión.

Por ultimo, el cliente enviar un flag ACK confirmando que la conexión se completo con éxito.

A partir de este punto comienza la transferencia de informacion.

En caso de que el servidor desee rechazar la conversación, debe responder al primer SYN con un mensaje RST indicando que la conexión no continuara.

Motivo por el cual se rechazaría una conexión

El principal motivo (a nivel de transporte) por el cual se rechazaría una conexión, es que el cliente desee conectarse a un puerto no abierto en el servidor. Es decir no hay ninguna aplicación que vaya a responder a dichos mensajes.

Transferencia de datos

Durante la etapa de transferencia de datos, una serie de mecanismos claves determinan la fiabilidad y robustez del protocolo. Entre ellos están incluidos el uso del número de secuencia para ordenar los segmentos TCP recibidos y detectar paquetes duplicados, checksums para detectar errores (como en UDP), asentimientos (ACK) y temporizadores para detectar pérdidas o retrasos y ventanas deslizantes para el control de flujo de datos.

ACK

Luego de cada mensaje enviado por alguna de las partes, se enviar un ACK confirmando la recepción del mensaje.

En caso de enviar y no recibir el mensaje ACK en un tiempo determinado, se considera no recibido y se vuelve a enviar. Si esto se repite multiples veces, se considera que la conexión se corto.

Cada implementación del protocolo TCP (es decir sistemas operativos de servidores y clientes) el tiempo de espera y cuantas veces se intenta reenviar la información.

Numero de secuencia

¿Que sucede si recibo un mismo paquete dos veces? (Debido a que tardo mucho en recibirse y la otra parte lo considero por perdido)

El numero de secuencia es un contador, que determinar el numero de paquete enviado. Si recibo dos paquetes con el mismo numero, significa que es el mismo.

Tambien puedo recibir el paquete 4 y luego el 6, el host debe saber que debe esperar la recepción del paquete 5. Ya que se pueden enviar multiples paquetes seguidos, y no es necesario esperar a recibir un ACK para volver a enviar otro.

Temporizadores y ventana deslizantes

Son otros elementos del protoclo TCP que buscan maximizar la eficiencia del protoclo. Buscan reducir la perdida de paquetes, envio de paquetes duplicados, etc. Y a su vez buscan maximizar la tasa de transferencia.

Fin de la conexión

La fase de finalización de la conexión utiliza una negociación en cuatro pasos, terminando la conexión desde cada lado independientemente.

Sin embargo, es posible realizar la finalización de la conexión en 3 fases; enviando el segmento FIN y el ACK en uno solo. Cuando uno de los dos extremos de la conexión desea parar su "mitad" de conexión transmite un segmento con el flag FIN, que el otro interlocutor asentirá con un ACK. Por tanto, una desconexión típica requiere un par de segmentos FIN y ACK desde cada lado de la conexión.

Una conexión puede estar "medio abierta" en el caso de que uno de los lados la finalice pero el otro no. El lado que ha dado por finalizada la conexión no puede enviar más datos pero la otra parte si podrá.

Comparativa UDP y TCP

UDP

Proporciona un nivel de transporte no fiable de datagramas, ya que apenas añade la información necesaria para la comunicación extremo a extremo al paquete que envía al nivel inferior. Lo utilizan aplicaciones como NFS (Network File System) y RCP (comando para copiar ficheros entre computadores remotos), pero sobre todo se emplea en tareas de control y en la transmisión de audio y vídeo a través de una red. No introduce retardos para establecer una conexión, no mantiene estado de conexión alguno y no realiza seguimiento de estos parámetros. Así, un servidor dedicado a una aplicación particular puede soportar más clientes activos cuando la aplicación corre sobre UDP en lugar de sobre TCP.[cita requerida].

TCP

Es el protocolo que proporciona un transporte fiable de flujo de bits entre aplicaciones. Está pensado para poder enviar grandes cantidades de información de forma fiable, liberando al programador de la dificultad de gestionar la fiabilidad de la conexión (retransmisiones, pérdida de paquetes, orden en el que llegan los paquetes, duplicados de paquetes...) que gestiona el propio protocolo. Pero la complejidad de la gestión de la fiabilidad tiene un coste en eficiencia, ya que para llevar a cabo las gestiones anteriores se tiene que añadir bastante información a los paquetes a enviar. Debido a que los paquetes para enviar tienen un tamaño máximo, cuanta más información añada el protocolo para su gestión, menos información que proviene de la aplicación podrá contener ese paquete (el segmento TCP tiene una sobrecarga de 20 bytes en cada segmento, mientras que UDP solo añade 8 bytes). Por eso, cuando es más importante la velocidad que la fiabilidad, se utiliza UDP. En cambio, “TCP asegura la recepción en destino de la información para transmitir”.

Ejemplo

El ejemplo mas claro de esto es Twitch, ya que me envia la transmicion de video y en caso de que haya mensajes que no me han llegado, no interesa enviarlos de vuelta para poder continuar viendo la totalidad del video, la prioridad es ver lo mas reciente.

A difrencia de Netflix, al cual puede interesarle saber que no me ha llegado una parte del video para poder reenviarla y no perder ninguna parte. Para este ultimo caso, lo mas optimo es utilizar TCP.
